# Refinement Techniques & Forcing Functions

## Prioritization Frameworks

### The "Desert Island" Test

Ask: "If this idea could only have ONE feature and nothing else,
which feature would it be?" Then: "OK, now add ONE more. Just one."
Continue until the user resists adding more. The resistance point
is the natural scope boundary.

### The Newspaper Headline Test

Ask: "If a journalist wrote one headline about this product, what
would it say?" If the headline requires "and" or "also" - the
idea isn't focused enough. A strong idea has one headline, not a
list of bullet points.

### The Regret Minimization Test

For each feature being considered for the cut: "If you ship
without this and it's successful, would you regret not including
it?" Most features fail this test. The ones that pass are the
ones that define the idea.

### The Competitor Test

Ask: "If a competitor launched tomorrow with ONLY the features
you're about to cut, would you be worried?" If no - those
features weren't differentiating. If yes - maybe they shouldn't
be cut.

### The Budget Halving Exercise

Present a hypothetical: "Your budget just got cut in half. Your
timeline just got cut in half. What survives?" Then do it again:
"Half again. Now what?" The features that survive two rounds of
halving are the true essentials.

## Elimination Exercises

### Kill Your Darlings

Identify the feature the user is most emotionally attached to.
Challenge it specifically. Often the most beloved feature is the
one most likely to be scope creep - it's kept because of
emotional investment, not because it's essential to the core
value proposition.

### The "Why Five Times" Drill

For any feature claimed as essential:
1. "Why is this essential?" → [answer]
2. "Why does that matter?" → [answer]
3. "Why is that important?" → [answer]
4. "Why would that be a problem?" → [answer]
5. "Why can't that be solved differently?" → [answer]

By the fifth "why," either the feature has proven its necessity
or the user has discovered it wasn't essential after all.

### The User Proxy Test

Ask: "Describe the ONE person who needs this most. Not a
demographic - a specific person with a name and a problem. Would
THAT person miss this feature if it were gone?" If the specific
user wouldn't miss it, cut it.

### The Inversion Test

Ask: "What if we built the OPPOSITE of what you're describing?
What would that look like?" Sometimes inverting the idea reveals
that the opposite is actually more interesting, or that certain
features only exist because of convention, not necessity.

## Conviction Testing Methods

### The Elevator Pitch Stress Test

After refinement, ask the user to pitch the refined idea in 30
seconds. Time it. If they can't do it without hesitation, stumbling,
or backtracking - the idea isn't clear enough yet. Record what
they stumbled on and investigate those elements.

### The "Would You Quit Your Job" Test

Not literally - but ask: "On a scale of 1-10, how excited are
you about what remains?" Anything below 7 means something was
lost in refinement that mattered, or the idea itself wasn't
strong enough. Both are valuable findings.

### The Overnight Test

If possible, suggest the user sleep on the refined version. Ask
them to return and describe the idea from memory without looking
at the brief. What they remember is what matters. What they
forget was probably chaff.

## Mode-Specific Techniques

### Ruthless Mode

- Open with: "Convince me this idea deserves to exist."
- Default position: every feature should be cut
- Demand evidence, not opinion
- Accept nothing on faith
- Use phrases like: "That's a nice-to-have, not a need-to-have"
  and "What breaks without this? Be specific."

### Firm Mode

- Open with: "Let's find what makes this idea special."
- Challenge but acknowledge good instincts
- Push on the weakest 2-3 elements hardest
- Use phrases like: "This is good, but is it essential?" and
  "I see the value - but does it earn its place in v1?"

### Socratic Mode

- Open with: "Walk me through this idea. I'll ask questions."
- Ask, don't tell
- Let silence do work - don't fill pauses
- Use phrases like: "What would happen if...?" and "How would
  you know if this feature was working?" and "What are you most
  uncertain about?"
